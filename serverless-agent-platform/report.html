Section 01

Executive Summary

What this platform is, why it was built, and what problem it solves

The Serverless Agent Platform is a fully autonomous, cloud-native AI execution system that requires zero local compute. Built and verified in July 2026, it enables a user to trigger complex AI-powered developer tasks from any device — including a mobile phone — and receive the results as a GitHub Pull Request automatically, with no manual intervention required.

The core problem it solves: How do you run AI agent workflows when you don't have a powerful PC available, don't want to pay for always-on cloud servers, and need results delivered in a developer-friendly format?

Core Value Proposition

Open a GitHub Issue from your phone. Two minutes later, receive a comment on that same issue with a link to a Pull Request containing the AI-generated response or code — all without touching a keyboard.

Key Innovations

  • Mobile-first design: the entire system is controllable from a smartphone
  • Dual-model routing: Groq classifies in milliseconds before expensive LLMs are invoked
  • Git-native delivery: results go directly into your repository as versioned markdown
  • Persistent memory: every run is logged to Supabase for future context and analytics
  • Zero infrastructure maintenance: GitHub Actions provides ephemeral compute for free
Section 02

How It Works

A plain-language walkthrough of the complete system from trigger to delivery

The platform is structured as a six-stage event-driven pipeline. Each stage is independent, fault-tolerant, and runs entirely in the cloud.

Stage 1 — The Trigger

The user creates a GitHub Issue with any natural language task as the title and body. GitHub immediately fires a webhook event. The issues: [opened] event in the workflow YAML catches this and wakes up an Ubuntu GitHub Actions Runner — a free, ephemeral Linux server in Microsoft Azure.

Example Trigger

Issue Title: "Build me a Python function that parses JSON from an API"
Issue Body:  "Use the requests library. Handle errors. Return a dict."
→ GitHub fires issues.opened webhook
→ Runner boots within ~15 seconds

Stage 2 — Environment Setup

The runner checks out the repository code, sets up Python 3.11, restores the pip dependency cache (speeds up installation from ~40s to ~5s after the first run), and installs all required packages from requirements.txt.

Stage 3 — Groq Classification (The Router)

Before invoking any expensive LLM, the task is sent to Groq's Llama 3.3 70B model with a strict zero-shot classification prompt. Groq returns either FRONTEND or GENERAL in under 200ms. This is the intelligence layer that ensures the right specialist model handles each task.

Stage 4 — Agent Execution

Based on the Groq classification:

  • FRONTEND tasks → routed to HuggingFace's Zephyr 7B (specialized for UI/CSS/React)
  • GENERAL tasks → routed to Google Gemini via the Antigravity SDK (complex reasoning)

The agent generates a response, which is written to output.txt in the runner's working directory.

Stage 5 — Memory Logging

Simultaneously, the complete run details (task, model used, result, status, timestamp) are inserted into the Supabase PostgreSQL database table agent_runs. This creates a full audit trail and enables future analytics.

Stage 6 — Git Delivery & Notification

The runner creates a unique branch (agent/issue-N), moves output.txt to docs/agent_responses/issue-N.md, commits it, and uses the GitHub CLI (gh) to open a Pull Request against main. Finally, it posts a comment on the original issue with the PR URL, completing the mobile notification loop.

Section 06

Core Components Deep Dive

Technical details of each major component in the system

6.1 — The Workflow File (issue-agent.yml)

The heart of the system. A 78-line YAML file that GitHub reads to know what to do when an issue is opened. It defines the trigger, permissions, environment, and all execution steps.

Key YAML Sections

on:
  issues:
    types: [opened]    # Only fires when NEW issue is created

permissions:
  contents: write         # Can push branches
  issues: write           # Can comment on issues
  pull-requests: write    # Can open PRs
  actions: read           # Can read workflow state

env:
  AGENT_TASK: "${{ github.event.issue.title }}\n${{ github.event.issue.body }}"
  GH_TOKEN: "${{ secrets.GITHUB_TOKEN }}"
  GITHUB_TOKEN: "${{ secrets.GITHUB_TOKEN }}"

6.2 — The Agent Script (agent.py)

A 112-line Python script that handles the complete AI pipeline: connecting to Supabase, calling Groq for routing, invoking the appropriate LLM, writing output, and logging the run. It is designed to be self-contained and fault-tolerant.

Agent Routing Logic

# Ultra-fast routing decision
category = groq_client.chat.completions.create(
    model="llama-3.3-70b-versatile",
    messages=[
        {"role": "system", "content": "Reply with FRONTEND or GENERAL only."},
        {"role": "user", "content": task_prompt}
    ],
    temperature=0,      # Deterministic output
    max_tokens=10       # Only need one word
)

if "FRONTEND" in category:
    # Route to HuggingFace Zephyr 7B
    run_huggingface_agent(task_prompt)
else:
    # Route to Gemini via Antigravity SDK
    async with Agent(config) as agent:
        response = await agent.chat(task_prompt)

Section 08

Security Model

How credentials are protected and what the attack surface looks like
⚠️
Important Note: During development, Supabase credentials were accidentally shared in a chat. All credentials (access token, anon key, publishable key) were immediately rotated. The new credentials are never stored in code — only in GitHub Secrets.

Secrets Management

All API keys are stored in GitHub Actions Secrets, which are encrypted at rest and injected as environment variables at runtime. They are never visible in logs, never in the repository, and auto-expire on certain events.

Secret NameServiceScope
GEMINI_API_KEYGoogle AIGemini/Antigravity SDK calls
GROQ_API_KEYGroq CloudLlama 3.3 routing calls
HUGGINGFACE_API_KEYHuggingFace HubZephyr 7B inference calls
SUPABASE_URLSupabaseDatabase endpoint URL
SUPABASE_KEYSupabaseAnon/public row-level access key
GITHUB_TOKENGitHubAuto-provided; PR/issue write access

Permission Principle of Least Privilege

The workflow only grants the exact permissions required for its actions — no more. Read-only access to actions metadata, write access to contents (for pushing branches), write access to issues (for comments), and write access to pull-requests (for opening PRs). No admin or delete permissions are granted.

Public Repository Considerations

ℹ️
Because the repo is public, any GitHub user can open an issue and trigger the agent. This means any user could consume your Groq/Gemini API quota. Mitigation: Add an actor check (if: github.actor == 'sumanthlucky3') to the workflow to restrict execution to your account only.

Pros, Cons & Limitations

An honest, comprehensive evaluation of the platform's capabilities and boundaries

✅ Strengths / Pros

  • Completely free — $0/month infrastructure cost
  • Zero local compute requirement
  • Mobile-first by design
  • Git-native output (versioned, reviewable, mergeable)
  • Dual-model routing optimizes cost and speed
  • Persistent memory via Supabase
  • Fully autonomous — no manual steps after issue creation
  • Secure — all secrets encrypted in GitHub
  • Easy to extend — add new routes, models, tools
  • No vendor lock-in — swap any LLM provider

❌ Weaknesses / Cons

  • 2-3 minute latency (not real-time)
  • GitHub Actions rate limits (2000 min/month free)
  • LLM quality varies — no review step by default
  • No multi-step reasoning (single-shot agent)
  • Public repo: anyone can trigger the agent
  • No file system access to existing codebase
  • Groq classifier can misroute edge cases
  • HuggingFace free tier has inference queues
  • Cannot execute or test the generated code
  • No real-time streaming of agent output

Appendix — Full Code Reference

Complete source code of all key files in the platform

A. .github/workflows/issue-agent.yml

name: 🤖 Autonomous Agent (Issue Trigger)

on:
  issues:
    types: [opened]

permissions:
  contents: write
  issues: write
  pull-requests: write
  actions: read

jobs:
  agent-response:
    runs-on: ubuntu-latest
    steps:
      - name: 📂 Checkout Repository Code
        uses: actions/checkout@v4

      - name: 🐍 Initialize Python Environment
        uses: actions/setup-python@v5
        with:
          python-version: '3.11'

      - name: ⚡ Restore Cached Python Packages
        uses: actions/cache@v4
        with:
          path: ~/.cache/pip
          key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}

      - name: 📥 Install Libraries
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt

      - name: 🤖 Run Cloud Agent
        env:
          GEMINI_API_KEY: ${{ secrets.GEMINI_API_KEY }}
          GROQ_API_KEY: ${{ secrets.GROQ_API_KEY }}
          HUGGINGFACE_API_KEY: ${{ secrets.HUGGINGFACE_API_KEY }}
          SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
          SUPABASE_KEY: ${{ secrets.SUPABASE_KEY }}
          AGENT_TASK: "${{ github.event.issue.title }}\n${{ github.event.issue.body }}"
        run: python agent.py

      - name: 📝 Commit Output and Create Pull Request
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          git config --global user.name "github-actions[bot]"
          git config --global user.email "github-actions[bot]@users.noreply.github.com"
          BRANCH_NAME="agent/issue-${{ github.event.issue.number }}"
          git checkout -b "$BRANCH_NAME"
          mkdir -p docs/agent_responses
          mv output.txt "docs/agent_responses/issue-${{ github.event.issue.number }}.md"
          git add "docs/agent_responses/issue-${{ github.event.issue.number }}.md"
          git commit -m "Agent Response to Issue #${{ github.event.issue.number }}"
          git push -u origin "$BRANCH_NAME"
          PR_URL=$(gh pr create \
            --title "Agent Response: ${{ github.event.issue.title }}" \
            --body "Automated response for issue #${{ github.event.issue.number }}" \
            --base main --head "$BRANCH_NAME")
          gh issue comment "${{ github.event.issue.number }}" \
            --body "Response ready: $PR_URL"
